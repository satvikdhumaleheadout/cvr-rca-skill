# Headout CVR-RCA — Business Domain Reference

This file is the shared business context for all CVR-RCA sub-skills and analyses.
Read this before interpreting any data, writing callouts, or making analytical
decisions. The definitions here are ground truth for all field values and segment
names that appear in `summary.json`.

---

## The Organisation — Headout

Headout is a tours, activities, and experiences marketplace operating across major
tourist cities worldwide. Revenue is generated by booking experiences
(museum entries, tours, activities, transfers) on behalf of customers, taking a
commission from the supply partner (the "take rate"). The core growth levers are:

- **Traffic** — bringing the right visitors to experience pages
- **CVR (Conversion Rate)** — turning visitors into confirmed bookings
- **AOV (Average Order Value)** — the value per booking
- **Completion Rate** — bookings that are fulfilled (vs cancelled)
- **Take Rate** — Headout's margin on each booking

CVR-RCA focuses specifically on the conversion lever.

---

## CE — Combined Entity

A **Combined Entity (CE)** is Headout's core unit of business reporting. It
represents a curated grouping of experiences within a specific geography —
typically a subcategory in a city (e.g., "Skip-the-Line Colosseum in Rome",
CE 167). Every CE has a `combined_entity_id` (the `ce_id` argument) and maps to
a `combined_entity_type` (collection, category, or subcategory).

CVR is measured per CE because demand, supply, and pricing dynamics are specific
to that geography × experience type. A CE can contain multiple individual
**experiences** (`experience_id`), each with their own variants and vendors.

## MB vs HO — The Two Distribution Channels

Headout distributes the same experiences through two front-ends:

| Code | Full name | What it is | Base CVR |
|------|-----------|------------|----------|
| **MB** | **Microbrand** | White-label city/theme microsites (e.g., `ticketslondon.com`, `museumsinflorence.com`). Traffic arrives primarily via SEO, affiliates, and partner referrals. | Lower — visitors are discovery-oriented, less committed |
| **HO** | **Headout** | The main `headout.com` platform. Traffic includes high-intent paid search (Google/Bing) and direct/email. | Higher — visitors are further down the purchase funnel |

**Why this matters for RCA:** MB typically converts at a meaningfully lower base
rate than HO. A shift in traffic share toward MB will depress overall CVR even
if per-session conversion rates in both brands are unchanged. Always distinguish
whether the drop is in **MB rates**, **HO rates**, or in **traffic mix** between
the two before diagnosing funnel issues.

Determined in data by URL: `LOWER(refined_page_url) LIKE '%headout%'` → HO,
otherwise → MB.

## Funnel Steps — Business Definitions

The CVR funnel is a 3-step product: **CVR = LP2S × S2C × C2O**.

| Step | Abbr | From event | To event | What the user is doing |
|------|------|-----------|---------|------------------------|
| **Listing Page to Select** | LP2S | Lands on a collection, category, theme, or experience listing page | Reaches the Select / Availability / Date-picker page | Browsing experiences, clicking through from the catalogue |
| **Select to Checkout** | S2C | On the select/seats/date-picker page | Reaches Checkout Started | Chose a date, variant, and seats — now entering payment flow |
| **Checkout to Order** | C2O | Checkout Started | Order confirmation page viewed | Completing payment and booking |

**What a drop in each step usually means:**

- **LP2S drop** — Users are landing but not clicking through to the select page.
  Common causes: price anchoring (displayed prices look too high vs competitors),
  listing page UX changes (images, sort order, filtering), low availability
  (no dates visible draws users away), or a mismatch between what the SEO/ad
  promised and what the listing shows. On MB, LP2S also covers microsite
  home/collection pages — a layout change or slow load on mobile is a frequent
  culprit.

- **S2C drop** — Users reached date/variant selection but didn't proceed to
  checkout. Common causes: no available dates for the selected period (supply
  issue), price shock at the date-picker stage (if base price is lower than
  variant price), confusing variant selection UX, or long lead times that
  dissuade last-minute bookers.

- **C2O drop** — Users started checkout but didn't complete the order.
  Decompose into C2A and A2O:
  - **C2A drop** = users abandoned before submitting payment → UX/pricing
    friction in the checkout form. DRI: Product/UX.
  - **A2O drop** = users submitted payment but order failed → payment gateway
    issue, fraud-check block, or live inventory failure (slot sold out between
    checkout start and payment completion). DRI: Payments team (gateway/fraud)
    or Ops/Engineering (inventory sync failure).

## Dimensions — What Each Segment Represents

### Device (`device`)

Derived from Mixpanel event properties. Three values:

| Value | What it is | Business context |
|-------|-----------|-----------------|
| `iOS Mweb` | iPhone/iPad browser (Safari, Chrome mobile) | Majority of MB traffic; checkout friction is highest here due to screen size and autofill issues |
| `Android Mweb` | Android browser | Similar profile to iOS Mweb; slightly lower share |
| `Desktop` | Any desktop/laptop browser | Higher intent, higher base CVR; mostly HO + paid search traffic |

A device-specific drop almost always points to a **UI/UX or performance
change** — mobile users abandon much faster when a page is slow or broken.

### Language (`language`)

Derived from the URL path (e.g., `/es/` → Spanish). Represents the **page
language the user is browsing in**, not the user's locale. English is the
default if no language code is in the URL.

Key interpretive notes:
- A drop in a specific non-English language often points to a **localization
  issue** (missing translations, broken layout for RTL languages), a
  **geo-specific pricing or availability change**, or a **marketing campaign
  in that market that drove lower-intent traffic**.
- A drop concentrated in English often points to a product or technical change
  (since most A/B tests and deploys target the default English experience first).

### Page Type (`page_type`)

Mixpanel event property on the listing/landing page event. Values differ by
brand:

**MB page types** (microsite landing pages):
| Value | What it is |
|-------|-----------|
| `Collection` | Curated experience list page (most common MB entry point) |
| `Theme` | Thematic grouping (e.g., "Best Rooftop Bars") |
| `Cruises Landing Page` | River/sea cruises-specific listing |
| `Hop-On Hop-Off` | HOHO bus-specific listing |
| `Airport Transfers` | Transfer-specific listing |
| `Content Page` | Editorial/blog-style pages — lowest purchase intent |
| `ShoulderPage` | City shoulder pages (nearby attractions) |

**HO page types** (headout.com):
| Value | What it is |
|-------|-----------|
| `Experience` | Single-experience detail page — highest intent |
| `Collection` | Category/collection browse page |
| `Category` | Top-level category browse |

A drop concentrated in **Collection or Content Page** types means the issue is
at the very top of the funnel — users are not even clicking from discovery pages
to experience-specific pages. A drop in **Experience** (HO) or the select page
means users with higher intent are still bouncing, which is a stronger signal of
a product or pricing problem.

## Channels — Acquisition Source Mapping

The `channel` dimension splits traffic by acquisition source. The key paid vs
organic distinction:

| Channel | Type | Intent level | Implication of a CVR drop |
|---------|------|-------------|--------------------------|
| Google Ads | Paid (Search) | High — user searched for the experience | Payment/checkout or pricing issue; these users have clear intent |
| Microsoft Ads (Bing) | Paid (Search) | High | Same as Google Ads |
| Facebook Ads (Meta) | Paid (Social/Display) | Medium — interest-based targeting | Could be audience mis-targeting or a creative → landing page mismatch |
| Affiliates | Referral | Medium | Could be traffic quality change from a specific affiliate partner |
| Things To Do (SEO) | Organic | Medium–Low | Often SEO traffic quality shift; check if ranking pages changed |
| Promotional Emails | CRM | Medium–High (existing users) | If CVR drops here, look at email targeting or offer mismatch |
| Transactional Emails | CRM | High (existing users) | Very high base CVR; a drop here is unusual and worth flagging |
| Others / Direct | Mixed | Variable | Catch-all; treat as directional only |

**Paid vs Organic framing**: If the channel mix shifts toward lower-intent
organic/affiliate traffic (e.g., because a paid campaign was paused), CVR will
fall even if all per-channel rates hold. That is a `channel_mix_dominant` case.

## Data Source — The Funnel Table

All funnel data comes from:
```
analytics_reporting.mixpanel_user_page_funnel_progression
```

Key columns relevant to CVR-RCA:

| Column | Meaning |
|--------|---------|
| `combined_entity_id` | CE identifier — filter on this (string comparison) |
| `experience_id` | Individual product within a CE — only set after user reaches select page |
| `is_microbrand_page` | `TRUE` if the user is on an MB microsite, `FALSE` for headout.com |
| `has_select_page_viewed` | LP → Select step: user reached the date/variant picker |
| `has_checkout_started` | Select → Checkout step: user started checkout |
| `has_order_attempted` | Checkout → Attempt: user submitted payment (C2A numerator) |
| `has_order_completed` | Attempt → Order: payment confirmed (A2O numerator) |

**Note on order confirmation:** `Order Confirmed` events often have missing
experience IDs (>75% of cases), so the pipeline uses `Order Confirmation Page
Viewed` as the proxy for a completed order. This is reliable for CVR measurement.

## Key Analytical Concepts

### Shapley Decomposition
LP2S × S2C × C2O is a 3-factor product. Shapley allocates the total ΔCVR
across factors by averaging their marginal contribution across all 6 orderings.
The three values always sum exactly to ΔCVR — there is no residual. This is why
you can say "LP2S explains 68% of the drop" with precision.

### Mix vs Conversion Effect
For each segment (MB/HO, channel):
- `mix_effect = Δshare × pre_rate` — what would have changed if only traffic
  share shifted (marketing/routing problem)
- `conversion_effect = pre_share × Δrate` — what would have changed if only
  the conversion rate shifted (product/funnel problem)
- `interaction = Δshare × Δrate` — the cross term (usually small)

When `conversion_effect` dominates → funnel problem.  
When `mix_effect` dominates → traffic routing problem.

### Traffic Non-Additivity
Users counted across dimension cuts (language, page_type, device) are not
additive — the same user can appear in multiple buckets. Each dimension has its
own COUNT DISTINCT subquery. Never sum user counts across dimensions or claim
"X% of users" across two cuts.

### C2O Sub-Decomposition
C2O = C2A × A2O. C2A measures users who attempt payment; A2O measures payment
success. Already computed in Stage 1 using `users_order_attempted` — no extra
BQ query needed.

## DRI Quick Reference

This table maps the most common finding to its most common DRI. It is a starting point, not a complete taxonomy — the confirmed root cause determines the actual DRI. Consult `hypothesis.md` and `actions.md` for the full cause → DRI mapping once the evidence has narrowed the diagnosis.

| Finding | Most common DRI | Other possible DRIs (see hypothesis.md) |
|---------|-----------------|----------------------------------------|
| LP2S drop, mobile web, sudden | Product (front-end / mobile web team) | Engineering (performance) |
| LP2S drop, gradual | Supply / Inventory (thin availability horizon) | Performance Marketing (SIS cap); BDM/Growth (competitive pricing); Content (content/media decay); Growth (assortment drift) |
| LP2S drop, specific language | Localisation / Marketing (geo campaign) | BDM (geo-specific pricing/availability); Performance Marketing (campaign routing) |
| S2C drop | Supply / Inventory (availability at date picker) or Product (UX) | Ops/BDM (vendor throttling, cut-off settings) |
| C2O: C2A drop | Product / UX (checkout form, pricing display) | Ops (custom fields, pax setup, booking fee); BDM (fee transparency) |
| C2O: A2O drop | Payments team (gateway, fraud rules) | Ops / Engineering (live inventory failure at fulfilment) |
| MB/HO mix shift | Marketing (traffic allocation, campaign budget) | Performance Marketing (campaign routing error) |
| Channel mix shift | Performance Marketing (paid spend change) |

## Table Schemas and Query Rules

This section is the reference for writing accurate custom queries. The nuances
here are not obvious from column names alone — skipping them produces plausible-
looking SQL that returns wrong numbers.

---

### `analytics_reporting.mixpanel_user_page_funnel_progression`

**What it is:** The primary funnel table. Every CVR analysis is built on this.

**Grain:** One row per user × session-starter event. Each row anchors to a single
landing page visit (the session-starter event) and carries forward-looking funnel
flags (`has_select_page_viewed`, `has_checkout_started`, etc.) derived by joining
downstream events for the same `user_id`. A user who visits three different pages
in separate sessions produces three rows — one per session-start.

**Critical counting rule:** Always `COUNT(DISTINCT user_id)` for traffic.
Never `COUNT(*)`. Never sum user counts across dimension cuts — a user who
appears in `language = 'French'` and `page_type = 'Collection'` is one person
counted in both cuts, not two. Each dimension needs its own `COUNT(DISTINCT user_id)`
subquery.

**Cross-session funnel attribution:** The funnel joins propagate forward in time
across sessions for the same user. Each downstream step (select, checkout, order)
is joined with the constraint `session_start_timestamp <= step_timestamp`, so a
user who landed Monday and booked Tuesday still gets `has_order_completed = TRUE`
on Monday's row. This means: filter on `event_date` for **session entry** timing,
not booking timing. Funnel rates over a date range always count users whose
session started in that window, regardless of when they converted.

**Key columns:**

| Column | Type | Notes |
|--------|------|-------|
| `user_id` | STRING | Mixpanel `distinct_id` — a device identifier, not a person. Incognito window = new user. |
| `event_date` | DATE | Date of the session-start event. **Partition key** — always filter on this for performance. |
| `session_id` | STRING | Unique session identifier, constructed as `CONCAT(session_number, '-', user_id)`. A new session is started when the acquisition source (campaign/channel) changes between consecutive events for the same user. |
| `event_timestamp` | TIMESTAMP | Timestamp of the session-starter event itself (the landing page hit). |
| `session_start_timestamp` | TIMESTAMP | Same as `event_timestamp` for session-opener rows — used as the anchor for downstream funnel joins. |
| `combined_entity_id` | STRING | CE identifier. **Already a STRING** — filter with `combined_entity_id = '<ce_id>'` (no CAST needed). Derived via 4-level priority: (1) URL-to-CE mapping table, (2) `collection_id` from event properties, (3) `category_id + ' - ' + city` for broad categories (4, 10, 12, 16, 18), (4) `sub_category_id + ' - ' + city` for all others. NULL if none could be resolved. |
| `combined_entity_mapping_source` | STRING | How `combined_entity_id` was derived: `'page_url_mapping'`, `'events_properties_collection_id'`, `'events_properties_category_id'`, or NULL. Useful for debugging unexpected CE assignments. |
| `experience_id` | STRING | Tour Group ID selected during funnel. **NULL until the user reaches the select page** — LP-level rows have no experience. |
| `experience_date` | DATE | Date of the experience selected during checkout. NULL if user never reached checkout. |
| `experience_time` | STRING | Time slot selected during checkout. NULL if user never reached checkout. |
| `lead_time_days` | INT64 | Days between checkout start and the experience date. NULL if user never reached checkout. |
| `is_microbrand_page` | BOOLEAN | **TRUE = MB microsite, FALSE = headout.com.** Derived as `LOWER(page_url) NOT LIKE '%headout%'`. This is the correct MB/HO flag to use — not `flow_type` (that column does not exist in this table). |
| `page_url` | STRING | Session-start URL, stripped of query params, URL fragments, and language path segments. |
| `previous_page_url` | STRING | Referrer URL. NULL for direct traffic or when referrer is unknown. |
| `page_type` | STRING | Type of the landing page: `'Microsite'`, `'Collection'`, `'Experience Page'`, `'Category Page'`. See Dimensions section for business meaning. |
| `page_sub_type` | STRING | Sub-type of the landing page. **Only populated for Microsite Page Viewed events.** NULL for all HO page types. |
| `language` | STRING | Full language name derived from the URL path (e.g., `/fr/` → `'French'`, no prefix → `'English'`). Represents the page language, not the user's locale. |
| `device_type` | STRING | Normalized device category. Exactly four values: `'iOS Mweb'`, `'Android Mweb'`, `'Other Mweb'`, `'Desktop'`. |
| `browsing_city` | STRING | City from which the user is browsing, from Mixpanel geo-IP. |
| `browsing_country` | STRING | Full country name from which the user is browsing (e.g., `'United Kingdom'`). Joined from a countries reference table. **Not an ISO code.** |
| `campaign_id` | STRING | Marketing campaign ID with numeric channel prefix stripped. NULL if user did not arrive via a tracked campaign. |
| `channel_id` | STRING | Marketing channel ID (numeric). |
| `channel_name` | STRING | Human-readable channel name (e.g., `'Google Ads'`, `'Facebook Ads'`, `'Affiliates'`). Derived from `channel_id` and referrer domain. |
| `advertising_channel_type` | STRING | Sub-channel for Google/Bing campaigns (e.g., `'Search'`, `'PAMX'`, `'Discovery'`). **Always filter `advertising_channel_type != 'PERFORMANCE_MAX'`** — this automated campaign type inflates traffic counts and distorts funnel metrics. NULL for non-paid channels; defaults to `'Others'` for unmapped paid campaigns. **Important:** This is joined from `ads_campaign_stats` using only the last 3 days of data — historical rows from campaigns that are no longer active may have NULL here even if they were a paid channel. |
| `select_page_viewed_timestamp` | TIMESTAMP | When user triggered 'Experience Selected' or 'Seats Page Viewed'. NULL if user never reached this step. |
| `checkout_started_timestamp` | TIMESTAMP | When user triggered 'Checkout Started'. NULL if user never reached checkout. |
| `order_attempted_timestamp` | TIMESTAMP | When user clicked 'Confirm And Pay'. NULL if user never attempted payment. |
| `order_completed_timestamp` | TIMESTAMP | When user viewed the Order Confirmation page (proxy for order completion). NULL if user never completed. |
| `has_select_page_viewed` | BOOLEAN | TRUE if `experience_id IS NOT NULL` — user reached the experience selection step. |
| `has_checkout_started` | BOOLEAN | TRUE if `checkout_started_timestamp IS NOT NULL`. |
| `has_order_attempted` | BOOLEAN | TRUE if `order_attempted_timestamp IS NOT NULL`. |
| `has_order_completed` | BOOLEAN | TRUE if `order_completed_timestamp IS NOT NULL`. **This is a proxy** — uses 'Order Confirmation Page Viewed' because the actual 'Order Completed' event lacks `experience_id` in >75% of cases. Reliable for CVR measurement. |

**Columns NOT in this table (common mistakes):**
- `flow_type` — does not exist; use `is_microbrand_page` instead
- `session_date` — does not exist; use `event_date`
- `experience_name`, `experience_city`, `experience_country`, `experience_region`, `business_market`, `business_region` — not in this table; join `dim_combined_entities` on `combined_entity_id` for those fields
- `browsing_country_code` — not in final table; `browsing_country` is the full name

**Standard filter pattern (use in all queries):**
```sql
WHERE combined_entity_id = '<ce_id>'
  AND event_date BETWEEN '<start>' AND '<end>'
  AND (advertising_channel_type IS NULL
       OR advertising_channel_type != 'PERFORMANCE_MAX')
```

**Standard MB/HO segmentation:**
```sql
CASE WHEN is_microbrand_page THEN 'MB' ELSE 'HO' END AS mb_ho
```

**Standard CVR query pattern:**
```sql
SELECT
    COUNT(DISTINCT user_id) AS users,
    COUNT(DISTINCT CASE WHEN has_select_page_viewed THEN user_id END) AS users_select,
    COUNT(DISTINCT CASE WHEN has_checkout_started THEN user_id END) AS users_checkout,
    COUNT(DISTINCT CASE WHEN has_order_attempted THEN user_id END) AS users_attempted,
    COUNT(DISTINCT CASE WHEN has_order_completed THEN user_id END) AS users_completed,
    SAFE_DIVIDE(
        COUNT(DISTINCT CASE WHEN has_order_completed THEN user_id END),
        COUNT(DISTINCT user_id)
    ) AS cvr

FROM `headout-analytics.analytics_reporting.mixpanel_user_page_funnel_progression`

WHERE combined_entity_id = '<ce_id>'
  AND event_date BETWEEN '<start>' AND '<end>'
  AND (advertising_channel_type IS NULL
       OR advertising_channel_type != 'PERFORMANCE_MAX')
```

### `analytics_features.product_rankings_features`

**What it is:** Daily feature data for the product ranking algorithm. Used in
CVR-RCA for price analysis (LP2S) and availability analysis (S2C).

**Grain:** One row per `query_id` × `product_id` × `event_date`. The
dbt staging view (`stg_product_rankings__features`) de-duplicates using
`ROW_NUMBER() OVER (PARTITION BY query_id, product_id, event_date)`.

**Column naming — raw table vs staging view:**

The raw source table (`analytics_features.product_rankings_features`) uses
`query_id` and `product_id`. The dbt staging view renames them:

| Raw table column | Staging view column | Notes |
|-----------------|---------------------|-------|
| `query_id` | `entity_id` | The combined entity ID (e.g., a collection ID or `'167'`) |
| `product_id` | `experience_id` | The experience/tour-group ID. Cast to STRING in the view. |
| `event_date` | `feature_date` | Date the features were computed. |

**For CVR-RCA queries, use the raw table directly** (`analytics_features.product_rankings_features`)
since the staging view is in a different project schema. Column names below
are for the raw table.

**Key columns (raw table):**

| Column | Type | Notes |
|--------|------|-------|
| `query_id` | STRING/INT | Combined entity ID. Cast to STRING when joining to funnel table: `CAST(query_id AS STRING) = combined_entity_id`. |
| `product_id` | STRING/INT | Experience ID. Cast to STRING when joining to funnel table: `CAST(product_id AS STRING) = experience_id`. |
| `event_date` | DATE | Date the features were computed. Filter on this. |
| `final_price_usd` | FLOAT64 | Listed price after any discount — what users see on the listing page. Use for price analysis. |
| `original_price_usd` | FLOAT64 | Price before discount. |
| `lev_score` | FLOAT64 | Listing Experience Value score — higher means better listing quality. |
| `days_to_first_available_date` | INT64 | Days from `event_date` to first available booking date. |
| `count_days_available_30d` | INT64 | Number of days with available inventory in the next 30-day window. Useful as availability proxy for S2C analysis. |
| `is_combo` | BOOLEAN | Whether this is a combo product. |
| `count_ratings` | INT64 | Total number of customer ratings. |
| `average_rating` | FLOAT64 | Average customer rating (1–5 scale). |
| `revenue_contribution_last30day` | FLOAT64 | **Ratio (0–1).** Revenue share of this experience within the entity over last 30 days. Never sum across rows. |
| `selects_contribution_last30day` | FLOAT64 | **Ratio (0–1).** Selects share of this experience within the entity. Never sum across rows. |
| `checkouts_contribution_last30day` | FLOAT64 | **Ratio (0–1).** Checkouts share. Never sum across rows. |
| `orders_contribution_last30day` | FLOAT64 | **Ratio (0–1).** Orders share. Never sum across rows. |

**All `*_contribution_*` columns are ratios (0–1), not counts.** They cannot
be summed across rows or dates — they represent within-entity share at a
point in time.

**Standard filter pattern:**
```sql
WHERE event_date BETWEEN '<start>' AND '<end>'
  AND final_price_usd IS NOT NULL
  AND final_price_usd > 0
```

**Joining to funnel table:**
```sql
FROM `headout-analytics.analytics_features.product_rankings_features` AS prf
JOIN `headout-analytics.analytics_reporting.mixpanel_user_page_funnel_progression` AS f
    ON CAST(prf.product_id AS STRING) = f.experience_id
    AND prf.event_date = f.event_date
```

### `analytics_reporting.dim_combined_entities`

**What it is:** One row per `combined_entity_id`. Use for metadata enrichment
of funnel results — CE name, type, geography, and planning attributes.

**Grain:** One row per `combined_entity_id`. No aggregation needed.

**Key columns:**

| Column | Type | Notes |
|--------|------|-------|
| `combined_entity_id` | STRING | Primary key. Matches the `combined_entity_id` in the funnel table. |
| `combined_entity_name` | STRING | Human-readable name of the CE. |
| `combined_entity_type` | STRING | **Explicitly stored** — `'collection'` (ID has no ` - ` separator), `'category'` (ID prefix is one of `4, 10, 12, 16, 18`), or `'subcategory'` (all others). |
| `country` | STRING | Country where the CE is located. Sourced from `dim_experiences`; may be NULL for sheet-only CEs. |
| `market` | STRING | Business market classification (e.g., `'SEA (MLY + IND + VN)'`, `'South America'`). |
| `region` | STRING | Business region classification. |
| `combined_entity_category` | STRING | High-level category grouping from the CE metadata sheet (e.g., `'POI/Attractions'`, `'Non-POI'`, `'Entertainment'`). |
| `combined_entity_subcategory` | STRING | Subcategory grouping for additional granularity. |
| `evolution_bucket` | STRING | Evolution bucket classification (planning use). |
| `management_type` | STRING | How the CE is managed. Resolved via 3-tier fallback: (1) latest `management_type_updated` event, (2) `collection_created` event, (3) sheet metadata. |
| `is_existing` | BOOLEAN | Whether this is a recognized existing CE. |
| `headout_status` | STRING | Current status on Headout (e.g., active/inactive). |
| `bucket_2025` | STRING | 2025 planning bucket classification. |

**`combined_entity_type` derivation logic** (derived from the ID pattern):
- ID does **not** contain `' - '` → `'collection'`
- ID prefix before `' - '` is one of `4, 10, 12, 16, 18` → `'category'`
- Everything else → `'subcategory'`

**Standard join pattern to enrich funnel results:**
```sql
LEFT JOIN `headout-analytics.analytics_reporting.dim_combined_entities` AS dce
    ON funnel.combined_entity_id = dce.combined_entity_id
```

### `analytics_reporting.dim_experiences`

**What it is:** One row per `experience_id`. Use for joining human-readable
experience names onto funnel results and for product-level price/rating metadata.

**Grain:** One row per `experience_id`. Primary key is `experience_id` (STRING).

**Key columns for CVR-RCA:**

| Column | Type | Notes |
|--------|------|-------|
| `experience_id` | STRING | Primary key. Matches `experience_id` in the funnel table. |
| `experience_name` | STRING | Human-readable product name. |
| `combined_entity_id` | STRING | Which CE this experience belongs to — same derivation as the funnel table's `combined_entity_id`. Use to join experiences → CE. |
| `combined_entity_name` | STRING | Human-readable CE name, derived alongside `combined_entity_id`. |
| `city` | STRING | Primary city of the experience. |
| `country` | STRING | Country of the experience. |
| `business_market` | STRING | Business market classification. |
| `business_region` | STRING | Business region classification. |
| `is_available` | BOOLEAN | **Live flag** — TRUE if `next_available_date >= CURRENT_DATE` (adjusted for city timezone). **Not reliable for historical analysis.** Use `product_rankings_features.count_days_available_30d` for historical availability. |
| `next_available_date` | TIMESTAMP | Next available booking date. Also a live snapshot. |
| `minimum_price` | FLOAT64 | Minimum price when inventory is available. |
| `listing_final_price` | FLOAT64 | Final listing price after discount. |
| `listing_original_price` | FLOAT64 | Original listing price before discount. |
| `average_rating` | FLOAT64 | Current average customer rating. |
| `count_ratings` | INT64 | Number of customer ratings. |
| `primary_collection_id` | STRING | Primary collection this experience belongs to. |
| `primary_category_id` | STRING | Primary category ID. |
| `primary_sub_category_id` | STRING | Primary subcategory ID. |

**`combined_entity_id` derivation** (same logic as funnel table, for consistency):
1. If `primary_collection_id` is present → use `primary_collection_id`
2. If category is one of `4, 10, 12, 16, 18` → `CONCAT(category_id, ' - ', city)`
3. Otherwise → `CONCAT(sub_category_id, ' - ', city)`

**Standard join pattern to enrich funnel results with experience names:**
```sql
LEFT JOIN `headout-analytics.analytics_reporting.dim_experiences` AS de
    ON funnel.experience_id = de.experience_id
```
